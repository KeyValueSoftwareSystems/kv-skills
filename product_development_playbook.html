Execution playbook

AI-assisted product development flow

A practical operating model for using AI tools, Claude Code-style skills, specialist subagents, deterministic checks, and human approvals to build backend and frontend features safely.

1. Purpose

This playbook is for putting AI-assisted product development into practice. It defines how a feature moves from human intent to product/technical plan, backend/frontend implementation, verification, human review, PR approval, and release.

What this is

A controlled SDLC workflow where AI accelerates planning, implementation, review and correction, while deterministic checks and humans control quality, risk and release.

What this is not

It is not a plan to create many independent bots that freely edit repos. The system uses one lead workflow, read-only specialist reviewers, bounded fix loops and explicit approval gates.

2. Operating model

Human approval AI work Deterministic verification Release

One lead workflow

The lead AI workflow plans, coordinates and summarizes. It should not allow every specialist to independently modify the repo.

Read-only specialists

Architecture, security, test-gap, API and UI-state reviewers should start as read-only subagents that return evidence-backed findings.

One implementation agent at a time

Backend and frontend implementation may run in parallel only when they touch different areas and the API contract is stable.

Bounded correction loop

The AI may attempt fixes, but only up to three times. After that it must stop and ask a human.

Deterministic proof

Lint, typecheck, tests, build, contract checks, Playwright and security tools are the proof. The model's claim is not proof.

Humans control risk

Humans approve intent, scope, API, DB, auth, risky changes, selected fixes, PR and release.

5. Backend complete flow

  • Plan before codeAI produces backend impact, API changes, DB/migration impact, tests, risk and rollback. Human approves before implementation.
  • Cartograph the repoBackend cartographer reads services, DB models, APIs, jobs, tests, infra and conventions. It writes context, not code.
  • Contract firstAPI contract is generated or updated before code. Include request, response, error shape, auth, permission and edge cases.
  • Implement approved scope onlyDomain logic, validation, persistence, API/controller, background jobs, tests, logs and docs.
  • VerifyRun lint, typecheck, unit tests, integration tests, build, migration check and contract validation.
  • ReviewRead-only architecture, security, test-gap and API reviewers return evidence-backed findings.

6. Frontend complete flow

  • Start from API + UX statesFrontend starts only after API contract is stable. UI state model must include loading, empty, success, validation error, API error, permission denied and retry/failure.
  • Component planIdentify pages affected, components reused, components added, API hooks, form schema, validation, analytics and tests.
  • ImplementTypes/API client, state management, component/page, form validation, error handling, accessibility, tests, Storybook and Playwright.
  • VerifyRun lint, typecheck, unit tests, build, Storybook/component checks and Playwright E2E.
  • UI state reviewRead-only reviewer checks missing states, keyboard accessibility, mobile handling, long names, error messages and retry behavior.

7. Verification and correction loop

Loop rule

The AI may fix failed checks, but the loop is bounded. After three failed attempts, or any risky change, it must stop and ask for human decision.

Never accept: Everything works.

Require: exact commands run, pass/fail status, failed command, root cause, files changed and remaining risks.

while checks_fail and attempts < 3:
    read exact failure
    identify smallest likely fix
    edit only related files
    rerun failing check
    record result

if targeted checks pass:
    run full verification

if attempts == 3 or risky change needed:
    stop and ask human

8. Human-in-the-loop gates

GateHuman confirmsAI must not do before approval
Plan approvalScope, assumptions, risks, backend/frontend splitStart implementation
API contract approvalEndpoints, DTOs, error shape, auth, permissionsLet frontend invent API or backend change contract
DB approvalMigration, indexes, rollback, data riskCreate or run risky migrations
Risky action approvalAuth, payment, production config, dependency upgrades, deletesExecute risky shell commands or broad refactors
Fix approvalWhich reviewer findings should be fixedAuto-fix every suggestion
PR approvalFinal diff, tests, risks and release readinessMerge or release

9. Parallel agent structure

Can run in parallel

Backend cartographer, frontend cartographer, architecture reviewer, security reviewer, test-gap reviewer, API contract reviewer, UI-state reviewer and documentation checker. These should be read-only.

Should not freely run in parallel

Refactor agent, migration agent, release agent and broad implementation agents. If they edit code, they need separate branches/worktrees and human-approved scope.

/plan-feature SKILL.md
---
description: Turn a feature request into an approved product and technical plan. Use before implementation. Do not edit code.
allowed-tools: Read, Grep, Glob, Bash
---

# /plan-feature

Goal: convert the human request into a clear implementation plan.

Rules:
- Do not edit code.
- Identify backend, frontend, API, DB, tests, security, rollout and risks.
- Ask questions when business rules are unclear.
- Explicitly list assumptions.
- Produce a human approval block before implementation.

Output:
1. Summary of request
2. In-scope / out-of-scope
3. Backend impact
4. Frontend impact
5. API contract impact
6. DB/migration impact
7. Test plan
8. Security/privacy risks
9. Rollback plan
10. Human approval required
/backend-implement SKILL.md
---
description: Implement an approved backend feature plan with tests and verification.
allowed-tools: Read, Grep, Glob, Bash, Edit, Write
---

# /backend-implement

Use only after plan and API/DB decisions are approved.

Before editing:
1. Read CLAUDE.md and AGENTS.md.
2. Read docs/ai/current-feature-context.md.
3. Read docs/architecture/overview.md.
4. Read docs/architecture/api-contracts/openapi.yaml if present.
5. List files you intend to change.
6. Ask for confirmation if DB, auth, payment, production config or dependencies are affected.

Implementation order:
1. Types/schema
2. Domain/service logic
3. Persistence
4. API/controller
5. Tests
6. Logging/metrics
7. Docs

Stop condition:
Do not say complete unless tests were run, failures are fixed or documented, changed files are summarized, and remaining risks are listed.

# /frontend-implement

Use only after the API contract is stable.

Required UI states:
- loading
- empty
- success
- validation error
- API error
- permission denied
- retry or failed operation

Implementation order:
1. Types/API client
2. State management
3. Component/page
4. Form validation
5. Error handling
6. Accessibility labels
7. Tests
8. Storybook if available
9. Playwright flow

Rules:
- Reuse existing components before creating new ones.
- Do not invent API shapes.
- Do not ignore TypeScript errors.
- Do not ship happy-path-only UI.

/fix-loop SKILL.md
---
description: Fix verification failures with a bounded loop. Maximum 3 autonomous attempts.
allowed-tools: Read, Grep, Glob, Bash, Edit
---

# /fix-loop

Maximum attempts: 3.

For each attempt:
1. Quote the exact failing command.
2. Summarize the error.
3. Identify the smallest likely fix.
4. Edit only files related to the failure.
5. Rerun the failing command.
6. Record the result.

After targeted pass:
Run full verification.

Stop and ask human if:
- DB migration is required but not approved
- API contract changes
- auth/permission behavior changes
- dependency upgrade is needed
- production config changes
- the same error persists after 3 attempts
- multiple design options are possible

# /human-review-pack

Output:
1. What changed
2. Why it changed
3. Files changed
4. API/DB/UI impact
5. Tests and commands run
6. Screenshots or Playwright trace links, if frontend
7. Risks and limitations
8. Review findings fixed
9. Review findings intentionally not fixed
10. Final human decision needed

12. Subagents and hooks

Read-only subagent template

name: architecture-reviewer
description: Read-only specialist reviewer for architecture drift and boundary violations.
tools:
  - Read
  - Grep
  - Glob
  - Bash
permissions:
  mode: read-only

instructions: |
  Do not edit files.
  Review the current change against documented architecture rules.
  Return only structured findings with evidence.

output_schema:
  summary: string
  findings:
    - severity: blocker | major | minor | suggestion
      area: backend | frontend | security | test | architecture
      file: string
      evidence: string
      recommendation: string
      safe_for_ai_fix: boolean

Hooks and guardrails template

# Pre-tool hook examples
# Block or ask before risky actions.

block_if_command_contains:
  - "rm -rf"
  - "git reset --hard"
  - "git push --force"
  - "DROP TABLE"
  - "terraform apply"
  - "kubectl apply"
  - "production"
  - ".env"

ask_human_before:
  - DB migration
  - auth/permission change
  - payment logic change
  - package upgrade
  - production config change
  - deleting files
  - external service integration

stop_hook_required_before_done:
  - changed files summarized
  - tests run or reason documented
  - risks listed
  - human review pack generated
