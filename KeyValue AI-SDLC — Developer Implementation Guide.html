Wiring the KeyValue AI-assisted SDLC

Developer implementation guide — from the product playbook to something developers run every day: one umbrella workspace per project, a shared skills pack, and a deterministic Microsoft Conductor workflow set across N repos and microservices.

Umbrella workspace · N repos keyvalue-skills pack Microsoft Conductor (YAML) Claude · Copilot · Cursor Scope: PRD/Figma ready → PR merge

Read the TL;DR first; the rest is detail. The aim is a controlled SDLC where AI accelerates design, implementation and QA while deterministic checks and humans control quality, risk and release.

0. TL;DR

  • Two things to build. A reusable keyvalue-skills pack (skills + Conductor workflows), and a per-project umbrella workspace that installs the pack and holds docs, tests, and N gitignored child repos.
  • Fix the process first. Independent FE/BE/microservice repos are cloned into one umbrella folder so an agent sees a whole feature, all tests live in one place, and stack up brings the entire stack up with one command.
  • Conductor runs the playbook. One main workflow composes sub-workflows: design, backend, frontend, qa. Routing is deterministic and version-controlled.
  • One skill, many front doors. The same skills are invoked by Conductor steps, by Claude Code, and by Cursor /commands. Behaviour lives in skills, not in the workflow.
  • Two ways to run it. A solo dev drives the whole main flow; or a lead runs design → design PR, and devs pull and run backend/frontend independently — as commands or as workflow runs.

The umbrella workspace is a per-project prerequisite, not part of what we build. Each team stands it up once per project. We only ship the tools and skills that make this easy — delivered in Phase 2. Phase 1 is the workflow set, which can run on a manually prepared umbrella.

1. The problem & the approach

Today's friction

  • Frontend, backend and multiple backend services live in separate, independent repos.
  • An agent designing one feature sees only one repo — it misses the cross-repo picture.
  • Skills are copied per repo and drift; no single place to manage them.
  • Integration and UI-automation tests are scattered; no suite covers the feature end to end.
  • No single command brings every service up locally.

The approach

  • Keep every repo independent (own branches, PRs, CI) — do not merge them.
  • Clone them, gitignored, into one umbrella workspace so the agent sees the whole feature.
  • Manage skills, docs and cross-repo test suites once, in the umbrella, from the shared pack.
  • Expose a single stack up / stack down to run the full stack locally.
  • Drive the playbook with Conductor — deterministic, version-controlled, swappable.

2. How the pieces fit

The shared pack is authored once; each project's umbrella installs it and clones its own N repos.

The pack is the product we maintain; the umbrella is where a project actually runs. Conductor steps and the slash-commands both call the same installed skills.

3. Prerequisites — adjust the process first

The SDLC workflow only works once these are in place. Do them in order, one per project.

P1
Create the umbrella workspace repo

A tracked git repo that becomes the agent's working surface for the whole feature. Child repos are cloned inside it but gitignored, so they stay independent.

Chosen: gitignored full clones + a workspace.yaml manifest. Alternatives considered: monorepo (collapses repo independence), git submodules (commit-pinned, painful coordinated branching), git subtree (collapses independence), symlinks (not git-aware, fragile).

P2
Install the keyvalue-skills pack

Pull the shared skills + Conductor workflows into the umbrella's .claude/skills and .cursor/commands. Managed in one place, updated by version.

Chosen: git-distributed pack, pinned by tag; workflows shareable via Conductor's Workflow Registry.

P3
Wire the full environment behind one command

The whole stack must come up and down with a single command — stack up / stack down. The agent owns the app stack only — never host prereqs or secrets (those are human pre-steps in SETUP.md).

Examples to choose from: Docker Compose (simple default), Tilt (live reload across many services), or Nix. Team picks per project; the workflow never edits stack files.

P4
Scaffold docs & centralise tests

Create the per-feature docs tree and move integration + UI-automation suites into the umbrella so one suite spans all repos.

Tooling: Playwright for web E2E/UI, Maestro (+ Maestro MCP) for mobile. Docs split: technical / functional / business, each with a folder per feature (auth, webhook, order-management…).

P5
Connect the source-of-truth MCPs

The design flow assumes PRD + Figma exist. Connect Jira / Confluence / Figma over MCP so the agent reads tickets, specs and designs directly.

4. The umbrella workspace

One workspace per project. Supports N child repos, including microservice fleets.

What it holds

  • Installed skills, the cross-repo test/integration and test/ui-automation suites, docs/, the workspace.yaml manifest, the kv and stack wrappers, contracts, and the gitignored child clones under codebase/.

What it does NOT do

  • It does not own the child repos' history, branches or CI — those stay independent. It is not a monorepo and not a spec engine. A file that says "done" is not proof; CI is.

workspace.yaml — the manifest (per-project, version-controlled)

project: ecommerce
repos:
  - name: frontend
    url: git@github.com:kv/ecommerce-frontend.git
    default_branch: main
    path: codebase/frontend
    run: { dev: "npm run dev", test: "npm test" }
  - name: order-service        # add as many services as needed
    url: git@github.com:kv/ecommerce-order-service.git
    default_branch: main
    path: codebase/order-service
    run: { dev: "./gradlew bootRun", test: "./gradlew test" }
  # - name: payment-service ...  (N repos supported)

contracts: contracts/            # shared, versioned cross-repo boundary

local_stack:
  type: docker-compose           # or: tilt | nix  (examples)
  up:          "docker compose -p {run_id} up -d"
  seed:        "./scripts/seed.sh"
  ready_check: "curl -sf http://localhost:{port}/health"   # MANDATORY
  ready_timeout: 180
  down:        "docker compose -p {run_id} down -v"         # always runs
  docs:        ./SETUP.md

5. Folder structure

Predictable paths so every agent and harness finds context, skills, tests and verification scripts in the same place.

  test/

6. Bootstrap skills

A small set of one-off skills — init-umbrella, migrate-repo, scaffold-docs, setup-integration-tests, setup-ui-automation, wire-local-stack — stand up the workspace and wire the prerequisites (each also exposed as a Cursor command and a Conductor step).

Implement as Phase 2. The SDLC workflows are Phase 1 and can run on a manually prepared umbrella; automate the prerequisites with these bootstrap skills once the core flow is proven.

7. The Conductor workflow set

One main workflow composes four reusable sub-workflows. Same shape as the playbook's 17 steps, made deterministic and version-controlled.

main.yaml (parent) Sub-workflow (runs standalone) AI / skill step Verify (script step) Reviewer / loop-back Human / git gate

How to read it: main.yaml is the parent workflow — it runs the four sub-workflows in order (design → backend ∥ frontend → qa) plus its own integrate step. design covers plan → approval → parallel cartographers → API contract → approval → Design PR. Inside backend / frontend the loop is: task split → execute → tests → verify; a failed verify loops back to execute (max 3 → human), and the reviewer can send it back with comments. Each sub-workflow also runs standalone (a /command or its own Conductor run); composition uses input_mapping and per-service for_each. The Design PR is a git gate between runs, not a Conductor gate.

View main.yaml — composition sketch 
name: kv-sdlc-main
provider: claude-agent-sdk      # skills + MCP discovered from CLI settings
steps:
  - id: design
    sub_workflow: design.yaml
    input_mapping: { feature: "{{ inputs.feature }}" }

  - id: branch
    type: script                # deterministic, token-free
    run: "kv branch --from design/affected_repos"

  - id: backend
    for_each: "{{ design.outputs.affected_services }}"   # fan-out per service
    sub_workflow: backend.yaml
    input_mapping: { service: "{{ item }}", contract: "{{ design.outputs.contract }}" }

  - id: frontend
    sub_workflow: frontend.yaml
    depends_on: [design]        # needs stable contract, runs alongside backend

  - id: qa
    sub_workflow: qa.yaml
    input_mapping: { acceptance: "{{ design.outputs.acceptance_criteria }}" }

  - id: integrate
    type: script
    run: "stack up -p {run_id} && kv test:e2e ; stack down -p {run_id}"   # trap-guaranteed down
View backend.yaml — bounded fix loop + env lifecycle 
name: kv-backend
inputs: [service, contract]
steps:
  - id: tasks       ; skill: "{{ skills.backend.tasks }}"     # default: backend-tasks
  - id: execute     ; skill: "{{ skills.backend.impl }}"      # may invoke Superpowers TDD
  - id: verify
    type: script
    run: "stack up -p {run_id} && kv test {{ inputs.service }} ; stack down -p {run_id}"
    on_fail: fix_loop
  - id: fix_loop
    skill: fix-loop
    max_iterations: "{{ workflow.fix_loop.max_attempts }}"   # cap; route to human on exhaustion
    route_back: verify
  - id: human_gate
    type: human                 # only reached after attempts are exhausted
    prompt: "Fix loop exhausted on {{ inputs.service }} — decide."

8. Two ways to run it

Same workflows, two operating modes. A team picks per feature.

A · Solo developer

One dev runs main end to end in their umbrella clone: design → branch → backend + frontend → qa → PRs. Best for single-repo or small features.

  • Single run, one owner, one branch set.
  • In-run human gates for plan / contract / risky actions.

B · Lead + devs (split by PR)

Lead runs design and raises the design PR to the umbrella. Once merged (contract locked), devs pull and pick up backend / frontend independently — either as Cursor/Claude /commands or as Conductor workflow runs (same skills behind both).

  • Separate devs = separate runs coordinated via git + contract.
  • Parallel implementation relies on the FE/BE/service repo split.

Conductor merges agent outputs, not git branches. Code merge and PR ordering stay in the umbrella + kv. There is no "two humans co-driving one run" — shared visibility comes from the dashboard URL (--web) or a hosted run.

10. Configuration — two files

Projects customise our layer by editing YAML — not by editing skills or third-party packs. We split config so a developer running a command standalone needs only skills.config.yaml, while Conductor reads both.

FileHoldsRead by
skills.config.yamlWhich skill (and its tool/framework) backs each slot.Commands and workflow
maestro.config.yamlPer-step models, fix-loop caps, enforced gates, env/isolation.Conductor workflow only

skills.config.yaml — skill selections (works standalone with commands)

# Which skill backs each slot. A dev running /backend uses only this file.
backend:
  tasks: backend-tasks
  impl:  backend-implement
  tdd:   superpowers          # or: none | custom
frontend:
  impl:  frontend-implement
qa:
  automation: qa-automation
  framework:  playwright       # web; maestro for mobile
reviewers: [architecture, security, test-gap, api, ui-state]

maestro.config.yaml — orchestration & enforcement (Conductor)

# Consumed by Conductor; not needed when running a skill via a command.
models:                        # per-step provider/model overrides
  design.hld:        claude-opus
  backend.fix_loop:  claude-sonnet
fix_loop:
  max_attempts: 3
gates:                         # enforced by script/CI steps, not by prompts
  coverage_threshold: 80
  require_tests_before_done: true
  freeze_approved_tests: true
env:
  isolation_key: "{run_id}"
  teardown: always
View backend-implement SKILL.md — reads config, composes a sub-skill 
---
description: Implement approved backend scope with tests and verification.
allowed-tools: Read, Grep, Glob, Bash, Edit, Write
---
# backend-implement

Before editing: read docs/technical/<feature>, the contract, list files to change,
ask if DB/auth/payment/prod/deps are touched.

For the implementation loop, read skills.config.yaml -> backend.tdd:
  - if `superpowers`: use the Superpowers RED-GREEN-REFACTOR skill.
  - if `custom`: use the project skill of that name.
  - if `none`: implement then add tests.

Stop condition: do not say complete unless tests ran, failures are fixed or
documented, changed files are summarised, and remaining risks are listed.
(Coverage / tests-before-done are enforced by the verify script step via workflow.config.)

11. Why Conductor — and the trade-offs

Why it fits the playbook

  • Deterministic — topology is declared, not discovered; orchestration consumes zero tokens. Ideal for a gated playbook.
  • Customisable — per-step models/providers, sub-workflows, parallel + for_each groups, route-back fix loops with caps.
  • Native UI — live dashboard DAG; attach screenshots; click a node for prompt, cost, output.
  • Per-team ownership — each lead edits their workflow YAML and model choices without touching skills.

Risks & how we hold them

  • Early-stage OSS (~v0.1.x). Don't rebuild it; keep workflow logic in skills + git so Conductor stays a swappable layer.
  • Verify before building: sub-workflow, provider/skill, loop/parallel semantics on the installed version.
  • Fallbacks if maturity bites: Composio AO, Temporal, LangGraph, Microsoft Agent Framework (GA, heavier).

Microsoft is actively promoting Conductor (OSS blog, May 2026), which signals continued investment — but treat it as a layer we can replace.
